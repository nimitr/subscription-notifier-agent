PROJECT WALKTHROUGH

Subscription Renewal Notifier

An autonomous agent that watches a Notion database, computes upcoming renewals, and pings a phone — all in ~270 lines of Python, running for free on GitHub Actions.

Python 3.12 GitHub Actions Notion API ntfy.sh Weekly cron

1 The idea

Subscriptions sneak up on you. Notion knows when they renew. The phone knows how to interrupt. Glue.

The problem

Recurring charges hit the card silently. Knowing on Monday what's about to charge this week (and the week after) makes them cancellable, not lamentable.

The data

A Notion database tracks each subscription: name, cost, next billing date. The billing date can even be a Notion formula — the script handles both shapes.

The delivery

ntfy.sh is a free push service: HTTP POST a body, the phone buzzes. No auth, no SDK, no app account — just a topic name.

2 Architecture

Four moving parts. One direction of data flow. No servers to run.

3 Execution flow

One run, top to bottom. Each Monday at 17:00 UTC (or on demand via the Actions tab).

①
windows
Compute two 7-day windows
Anchor on today: days 1–7 = "next 7", days 8–14 = "the week after".
②
query
Query Notion
Single paginated /databases/{id}/query over the full 14 days.
③
extract
Flatten properties
Pull name, due, cost out of Notion's nested shape.
④
partition
Split into two buckets
Mutually exclusive — nothing counted twice. No-date rows fall into the near bucket.
⑤
notify
POST to ntfy
ASCII title header + UTF-8 body. Phone buzzes ~1s later.

4 The two windows

Why rolling 7+7 instead of calendar weeks/months? No edge cases — the same partition works every day of the year.

today+7d+14d
Next 7 days
Week after (8–14)

Why not calendar months? Run during the last week of a month and the "later this month" bucket is always empty. Run during the first week and it's enormous. Rolling windows smooth that out and let the same code path work every Monday — no special "are we at the end of the month?" branch.

5 Code walkthrough

The script is one file. Here's what each notable piece does and why.

week_bounds removed → upcoming_windows(today) windowing

Returns the two boundary dates for the rolling 7+7 partition. Pure function, no I/O — trivially testable.

def upcoming_windows(today: date) -> tuple[date, date]:
    week1_end = today + timedelta(days=6)   # inclusive end of window 1
    week2_end = today + timedelta(days=13)  # inclusive end of window 2
    return week1_end, week2_end
query_notion(...) Notion HTTP + pagination

Posts a compound AND filter (on_or_after + on_or_before) and loops on has_more/next_cursor until every matching page is collected. Sorted ascending by date in the request, so no client-side sort is needed.

while True:
    payload = dict(base_payload)

extract_date(prop) Notion shape juggling

Notion's API returns dates two different ways depending on whether the column is a plain date or a formula. This function papers over both shapes — added after a real-world debug session where the filter matched rows but the date came out empty.

if prop.get("type") == "formula":
    f = prop.get("formula") or {}
    if f.get("type") == "date":
        return (f.get("date") or {}).get("start")
    return None
return (prop.get("date") or {}).get("start")
extract_cost(prop, cost_type) dual-mode cost parsing

Cost columns can be number (straight float) or rich_text (best-effort parsed via regex strip). Returns a (numeric, display) tuple where exactly one side is meaningful — the display string survives if parsing fails so the user always sees something.

partition loop in main() classification

Splits flat rows into the two buckets. The buckets are mutually exclusive by construction (the elif guarantees it).

for it in items:
    d = parse_iso_date(it["due"])
    if d is None:
        next_7.append(it)         # default no-date rows to near bucket
        continue
    if today <= d <= week1_end:
        next_7.append(it)
    elif week1_end < d <= week2_end:
        week_after.append(it)
    else:
        next_7.append(it)         # shouldn't happen, but don't drop
send_ntfy(...) HTTP header gotchas

ntfy uses HTTP headers for title/priority/tags, not JSON. Headers must be latin-1 encodable, so the title is constrained to ASCII — em-dashes (—) crash requests here. The body is the raw POST data, explicitly UTF-8 encoded, so unicode bullets and dashes work fine in the body.

build_message(next_7, week_after, currency) presentation

Composes the ntfy title + body. Adapts to four states: both buckets populated, only the near bucket, only the far bucket, or completely empty. The empty case sends a low priority message so it doesn't buzz the phone.

8 The workflow file

Twenty lines of YAML. Cron + manual trigger + three steps.

on:
  schedule:
    # Cron is UTC, no DST. 17:00 UTC = 10:00 AM PDT Mondays.
    - cron: "0 17 * * 1"
  workflow_dispatch:            # manual "Run workflow" button

jobs:
  notify:
    runs-on: ubuntu-latest
    timeout-minutes: 5
    env:
      DATE_PROPERTY: "Next Billing Date"
      COST_TYPE: "number"
      CURRENCY: "$"
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with: { python-version: "3.12", cache: "pip" }
      - run: pip install -r requirements.txt
      - env:
          NOTION_TOKEN: ${{ secrets.NOTION_TOKEN }}
          NOTION_DATABASE_ID: ${{ secrets.NOTION_DATABASE_ID }}
          NTFY_TOPIC: ${{ secrets.NTFY_TOPIC }}
        run: python subscription_notifier.py

9 Notable design decisions

No state between runs

Each run is a pure function of "today's date" + the current Notion DB. No checkpoints to corrupt, no "did I send this already?" bookkeeping. If a run is missed, the next one still tells the truth.

Single dependency

Just requests. The Notion SDK was avoided — two HTTP endpoints don't justify the dependency footprint, and the bare API is well-documented enough to use directly.

Fail loud, not silent

Every HTTP error calls raise_for_status(). A bad token, a moved database, a renamed column — anything wrong becomes a red Actions run in GitHub, not a silent skipped notification.

Tolerant extraction

Cost columns can be number or text. Date columns can be plain or formula. Missing dates default to the near bucket. The script reads imperfect input and never crashes on a single weird row.
